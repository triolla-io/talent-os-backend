End-to-end automated flow: from Mailgun webhook to a scored candidate record in the database.

📬
Webhook Intake
POST /webhooks/email
Mailgun sends the inbound email as multipart/form-data. The request passes through two guards before reaching the handler.
ThrottlerGuard — 100 req / 60s MailgunAuthGuard — HMAC signature Zod payload validation
🔁
Idempotency Guard
Check & Insert — emailIntakeLog
Prevents duplicate processing if Mailgun retries the webhook or a crash left a job mid-flight.
Decision Logic
status = 'pending' → Re-enqueue (previous attempt crashed)
status = 'completed' / 'spam' → Return 200, skip (already handled)
not found → INSERT new row with status='pending', continue
📦
Queue
Enqueue to BullMQ — "ingest-email"
The email payload (metadata only — CV file blobs stripped from the DB row) is pushed to Redis. BullMQ manages retries automatically.
3 attempts max Exponential backoff — 5s base Lock renewed every 5s API returns 200 immediately
Worker Process picks up job ↓
🚫
Phase 0
Spam Filter
Fast heuristic check before any expensive processing. Keyword-based rules on subject and body.
Spam Rules
no attachment + body < 100 chars → Mark spam, stop pipeline
spam keyword + no attachment → Mark spam, stop pipeline
spam keyword + has attachment → Flag as suspicious, continue (AI warned)
📄
Phase 1–2
Text Extraction — PDF & DOCX
Attachment files (base64 in the normalized payload) are decoded and parsed into plain text. Combined with the email body to form the full CV context.
PDF files
pdf-parse library
Extracts text layer from PDF
DOCX files
mammoth → HTML → plain text
Converts Word format to stripped text
Email body prepended Each file demarcated with header
☁️
Phase 5
Upload CV to Cloudflare R2
The largest PDF or DOCX attachment is uploaded to R2 before AI processing. This ordering is intentional — if AI extraction fails, the raw CV file is already persisted and can be recovered. The returned R2 key is stored on the candidate in Phase 7.
Key: cvs/{tenantId}/{messageId}.pdf Explicit ContentType set Failure → BullMQ retries entire job
🤖
Phase 4
AI Data Extraction — GPT-4o-mini
The full CV text is sent to OpenRouter. The model returns a structured JSON with the candidate's details.
Extracted Fields
full_name · email · phone
current_role · years_experience
location · skills[] · ai_summary
source_hint · source_agency
Fallback Strategy
Attempt 1–2: AI fails → BullMQ retry
Attempt 3 (final): try deterministic parser
(first line = name, regex email/phone)
Still fails → mark failed, stop
👥
Phase 6
Dedup Check + Candidate Insert / Upsert
Phone number exact match via regexp_replace(phone, '[^0-9]', '', 'g') identifies known candidates. An advisory lock on the phone value serializes concurrent workers.
Dedup Algorithm & Action
phone matches exactly (digits-only comparison) → Confidence 1.0 → INSERT new candidate row + CREATE DuplicateFlag linking to existing (first-submission preserved)
no phone in CV (phone_missing) → INSERT new candidate + CREATE DuplicateFlag (confidence=0, matchFields=['phone_missing']) for HR review
no match → INSERT new candidate record
Atomic transaction pg_advisory_xact_lock on phone hash
🔗
Phase 7a
Detect Matching Job(s) from Email Text
Scans the email subject and body for job shortIDs (e.g. "103"). All open jobs are fetched and their shortIDs are regex-matched against the email text.
Queries all open jobs first Word-boundary regex per shortId Multiple jobs can match one email
📊
Phase 7b
Enrich Candidate + Multi-Job AI Scoring
Candidate fields (role, skills, CV text, R2 key) are written. Then, for each matched job, an Application is created and GPT-4o-mini scores the fit.
DB Writes
UPDATE candidates (fields + cvFileUrl)
UPSERT application per job
CREATE CandidateJobScore per job
UPDATE candidate.aiScore (max score)
Scoring Output
score: 0–100
reasoning: text explanation
strengths: string[]
gaps: string[]
✅
Complete
Candidate Ready — emailIntakeLog.status = "completed"
The candidate appears in the recruiter dashboard with a score, summary, and CV link. If no job was detected, they appear in the unassigned queue.

Candidates API

HTTP endpoints for reading and managing candidate records, stage transitions, and manual adds.

GET /candidates
List with search & filters
Query Params
q — full-text search (name / email / role)
job_id — filter by job
filter=duplicates — only unreviewed dups
unassigned=true — no job yet
Response
candidates[] — full profiles
total — count
Includes: hiring stage, job title, duplicate flags
GET /candidates/counts
Dashboard counts
Returns { total, duplicates, unassigned } — lightweight aggregate for dashboard alert badges. Declared before /candidates/:id in the router to avoid route shadowing.
GET /candidates/:id
Full candidate detail
Returns candidate profile + all applications + AI scores per job + stage summaries + duplicate flags
POST /candidates
Manual candidate add (multipart/form-data)
1. Validate email uniqueness within tenant
2. Upload CV file to Cloudflare R2
3. Generate AI summary (non-blocking — continues even if AI fails)
4. Create Candidate + Application in atomic transaction
PATCH /candidates/:id
Update profile & job assignment
Job Assignment Logic
same job_id → Update profile fields only, no new Application
different job_id → Create new Application + trigger AI scoring for new job
no job_id currently → Assign job: create Application + score
PATCH /candidates/:id/stage
Kanban stage move
Atomically updates candidate.hiringStageId + application.jobStageId to the new stage.
POST /candidates/:id/stages/:stage_id/summary  &  /advance
Stage notes & advance
/summary — Upsert a text note for this candidate at this stage.
/advance — Save the summary AND move the candidate to the next enabled stage in one call.
GET /candidates/:id/cv-url
Presigned download URL
Calls Cloudflare R2 to generate a time-limited presigned URL (1 hour expiry). The client uses this URL to download the CV directly from R2.
POST /candidates/:id/reject
Sets candidate.status = 'rejected' and all linked applications to stage = 'rejected'.
DELETE /candidates/:id
Hard delete. Removes duplicate flags, nullifies intake log reference, cascades to applications and scores.

Jobs API

Manage job listings including their hiring pipeline stages and screening questions.

POST /jobs
Create job — atomic with stages & questions
1. Auto-generate shortId: advisory lock on tenant → query MAX(short_id) → increment (starts at 100)
2. Default hiring stages created if none provided: Application Review → Screening → Interview → Offer → Hired → Rejected → Pending Decision → On Hold (8 stages; Hired, Rejected, Pending Decision, On Hold are disabled by default)
3. Screening questions created (if provided)
4. All created atomically in a single transaction
GET /jobs
List all jobs with optional status filter (open / closed). Returns job metadata and candidate count.
GET /jobs/:id
Full job detail including hiringStages[], screeningQuestions[], and candidate count.
PUT /jobs/:id
Full replace of job fields, hiring stages, and screening questions.
DELETE /jobs/:id
Soft delete — sets status = 'closed'. Candidates and applications are preserved.
🔑 How Job ShortIDs Work
Each job gets a numeric shortId (e.g. 103) scoped to the tenant. When a candidate emails their CV, the system scans the email text for these shortIds using word-boundary regex. A match links the candidate to that specific job and triggers AI scoring for it.

Example: Email subject "Application for role 103" → candidate is scored against Job #103.
