2. Create `src/health/health.service.ts`.

IMPORTANT — Redis health check MUST reuse the existing BullMQ/ioredis connection rather than opening a new connection per check. Creating a new Redis connection on every health check is wasteful and can exhaust connection limits under monitoring.

Locate where the BullMQ queue connection is registered (likely in the ingestion or queue module). Inject that existing connection (or use `Queue.client` from BullMQ) into HealthService. If BullMQ exposes the underlying ioredis client via `Queue.opts.connection` or a registered `IORedis` provider, inject that. The exact approach depends on what the codebase exposes — read the module files first.

If no injectable ioredis instance exists, register one as a provider in HealthModule and share it with BullMQ rather than creating a new transient connection in checkRedis().

Pattern to follow:
```typescript
import { Injectable } from '@nestjs/common';
import { PrismaService } from '../prisma/prisma.service';
import { InjectQueue } from '@nestjs/bullmq';
import { Queue } from 'bullmq';

export interface HealthCheckResult {
  status: 'ok' | 'degraded';
  checks: {
    database: 'ok' | 'fail';
    redis: 'ok' | 'fail';
  };
  uptime: number;
}

@Injectable()
export class HealthService {
  constructor(
    private readonly prisma: PrismaService,
    // Inject existing BullMQ queue to access its underlying Redis connection
    // Replace 'ingestion' with the actual queue name used in this project
    @InjectQueue('ingestion') private readonly queue: Queue,
  ) {}

  async check(): Promise<HealthCheckResult> {
    const [dbOk, redisOk] = await Promise.all([
      this.checkDatabase(),
      this.checkRedis(),
    ]);

    const status = dbOk && redisOk ? 'ok' : 'degraded';

    return {
      status,
      checks: {
        database: dbOk ? 'ok' : 'fail',
        redis: redisOk ? 'ok' : 'fail',
      },
      uptime: Math.floor(process.uptime()),
    };
  }

  private async checkDatabase(): Promise<boolean> {
    try {
      await this.prisma.$queryRaw`SELECT 1`;
      return true;
    } catch {
      return false;
    }
  }

  private async checkRedis(): Promise<boolean> {
    try {
      // Use the existing BullMQ queue's Redis client — no new connection created
      const client = await this.queue.client;
      await client.ping();
      return true;
    } catch {
      return false;
    }
  }
}
```

If the queue name or injection pattern differs from what you find in the codebase, adapt accordingly. The key constraint: do NOT create a new `new Redis(url)` inside checkRedis(). Use the existing connection.

In `src/ingestion/ingestion.processor.ts`, after nestjs-pino is configured (Task 2), inject the pino logger rather than using the NestJS `Logger` class for lifecycle events.

IMPORTANT: NestJS Logger's second parameter is `context?: string` (a string), NOT a structured object. Passing `{ jobId, jobName }` as the second argument to `this.logger.log()` will produce unstructured output or a TypeScript error, violating D-24. Instead, use the pino `Logger` directly for lifecycle events.

Inject pino Logger from nestjs-pino:
```typescript
import { Logger } from 'nestjs-pino';
// Add to constructor:
// private readonly pinoLogger: Logger
```

Add lifecycle logging at these points inside the `process()` method using the pino interface (`logger.info({...}, 'message')` — structured object first, message string second):

1. At the very start of `process()` (before any other code):
```typescript
this.pinoLogger.assign({ jobId: job.id, jobName: job.name, tenantId: this.config.get('TENANT_ID') });
this.pinoLogger.info({ jobId: job.id, jobName: job.name, tenantId: this.config.get('TENANT_ID') }, 'Job started');
```

Or, if using pino-logger's log method directly:
```typescript
this.pinoLogger.log({ jobId: job.id, jobName: job.name, tenantId: this.config.get('TENANT_ID') }, 'Job started');
```

2. At the end of successful processing (where `processingStatus: 'completed'` is set):
```typescript
this.pinoLogger.log({ jobId: job.id, jobName: job.name, tenantId: this.config.get('TENANT_ID') }, 'Job completed');
```

3. In each catch block where an error is re-thrown (where `processingStatus: 'failed'` is set):
```typescript
this.pinoLogger.error({ jobId: job.id, jobName: job.name, tenantId: this.config.get('TENANT_ID'), error: err instanceof Error ? err.message : String(err) }, 'Job failed');
```

The key difference from the NestJS Logger API:
- NestJS Logger: `this.logger.log('message', optionalContextString)`
- Pino Logger:   `this.pinoLogger.log({ structured, fields }, 'message')`

These are NOT interchangeable. Use the pino form for all three lifecycle calls.

If the existing class already has `private readonly logger = new Logger(IngestionProcessor.name)` for other logging, keep it for those existing calls. Add `pinoLogger` as a separate injected dependency only for the structured lifecycle events.

- Add startup log: `logger.log('BullMQ Worker started');`

Do NOT break existing test mocks. The pino Logger injection adds a new constructor parameter — verify existing unit tests for IngestionProcessor mock the new dependency or use `{ provide: Logger, useValue: { log: jest.fn(), error: jest.fn() } }`.
