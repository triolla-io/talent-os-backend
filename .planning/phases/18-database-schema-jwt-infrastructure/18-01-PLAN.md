## Schema Context

Current Prisma schema has Tenant model; this plan renames to Organization and adds User and Invitation models.

From existing schema.prisma:
- Tenant model: id (UUID), name (text), createdAt
- All downstream models (Job, Candidate, Application, etc.) reference Tenant via tenantId field

## Auth Design (from spec/auth-rules.md)

- Owners sign up via Google OAuth; auth_provider = 'google' (AUTH-001)
- Invited users join via Magic Link; auth_provider = 'magic_link' (AUTH-004)
- NO password-based auth anywhere in the system
- 4 roles: owner (org creator, immutable), admin, member, viewer
- Invitations: role dropdown shows admin/member/viewer only (owner cannot be invited)

## Configuration Context

From CLAUDE.md:
- Constraints: TypeScript only, Prisma 7, PostgreSQL 16
- Conventions: text fields + CHECK constraints over ENUMs, no binary blobs, @updatedAt for timestamps
- Tech: @nestjs/config for ConfigModule, Zod for validation
- DB naming: snake_case in schema, camelCase in Prisma model

**JwtPayload interface** (role values must match spec/auth-rules.md AUTH-006):

```typescript
export interface JwtPayload {
  userId: string;
  organizationId: string;
  role: 'owner' | 'admin' | 'member' | 'viewer';
}
```

**JwtService implementation:**

```typescript
import { Injectable, UnauthorizedException } from '@nestjs/common';
import { ConfigService } from '@nestjs/config';
import * as jwt from 'jsonwebtoken';

export interface SignOptions {
  expiresIn?: string | number;
}

@Injectable()
export class JwtService {
  private readonly jwtSecret: string;

  constructor(private configService: ConfigService) {
    this.jwtSecret = this.configService.get<string>('JWT_SECRET');
    if (!this.jwtSecret) {
      throw new Error('JWT_SECRET is not defined in environment variables');
    }
  }

  sign(payload: JwtPayload, options?: SignOptions): string {
    return jwt.sign(payload, this.jwtSecret, {
      expiresIn: options?.expiresIn ?? '15m',
    });
  }

  verify(token: string): JwtPayload {
    try {
      return jwt.verify(token, this.jwtSecret) as JwtPayload;
    } catch (error) {
      if (error instanceof jwt.TokenExpiredError) {
        throw new UnauthorizedException('Token has expired');
      }
      throw new UnauthorizedException(`Invalid token: ${(error as Error).message}`);
    }
  }

  signAccessToken(payload: JwtPayload): string {
    return this.sign(payload, { expiresIn: '15m' });
  }

  signRefreshToken(payload: JwtPayload): string {
    return this.sign(payload, { expiresIn: '7d' });
  }
}
```

**JWT payload maps to:**
```json
{
  "sub": "...",    // Note: jsonwebtoken uses "sub" for userId if we map it, or just include userId directly
  "userId": "user-uuid",
  "organizationId": "org-uuid",
  "role": "owner|admin|member|viewer",
  "iat": 1234567890,
  "exp": 1234571490
}
```

**Notes:**
- No `permissions` array in JWT payload — permissions are derived from role in Phase 21 guard layer
- Fail fast if JWT_SECRET missing (constructor throws)
- No database interaction in JwtService (pure crypto)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Google OAuth → API | OAuth token from Google is validated by Google's SDK; user identity comes from verified Google account |
| Magic Link email → API | Token in URL is validated against invitations table; one-time use enforced by status='accepted' |
| API → Database | SQL injection via Prisma ORM (parameterized queries); org isolation via FK constraints |
| Environment → App | JWT_SECRET injected via environment variables (trusted if env is secure) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-01 | Spoofing | JwtService | Mitigate | Use strong JWT_SECRET (min 32 chars); validate at app startup (fail-fast) |
| T-18-02 | Tampering | Database schema | Accept | PostgreSQL enforces CHECK constraints and FK integrity; no app-level bypass possible |
| T-18-03 | Spoofing | Invitation token | Mitigate | Token must be unique across all invitations; one-time use enforced by status field; Phase 20 validates token + expiry |
| T-18-04 | Elevation of Privilege | Role field | Mitigate | Role stored as text with CHECK constraint limiting values; 'owner' cannot be set via invite (invitation role CHECK excludes 'owner') |
| T-18-05 | Elevation of Privilege | auth_provider | Mitigate | auth_provider CHECK constraint prevents invalid values; Phase 19 sets provider based on auth flow, not user input |
| T-18-06 | Information Disclosure | JWT token | Accept | Phase 19 will enforce HTTPS; JWT payload contains only userId/orgId/role (no sensitive data) |

**Summary:** Phase 18 is infrastructure-only (no API endpoints). Biggest risk is JWT_SECRET exposure (mitigated by secure env storage and fail-fast validation). DB constraints enforce role integrity and prevent invitation privilege escalation.

</threat_model>
